## اصلاح ۱ — آشتی با سیستم توکن موجود (جایگزین §۷ سند اصلی)

سند اصلی مقادیر پیکسلی‌ای تعیین کرده بود که در مقیاس تایپوگرافی `globals.css` وجود
ندارند و گیت `bun run lint:design` آن‌ها را رد می‌کند. جدول نهایی و لازم‌الاجرا:

| نقش | سند اصلی | توکن نهایی پروژه | اندازه واقعی |
|---|---:|---|---:|
| عنوان صفحه | 24px | `text-xl font-bold` | 20px |
| عنوان بخش | 16px | `text-lg font-bold` | 16px |
| عنوان کارت | 14px | `text-base font-semibold` | 14px |
| متن اصلی | 14px | `text-base` | 14px |
| Label | 13–14px | `text-base font-medium` | 14px |
| Metadata / timestamp | 12px | `text-sm` | 12.5px |
| عدد KPI | 28–32px | `text-2xl font-bold tabular-nums` | 26px |
| متن جدول | 13px | `text-sm` تا `text-base` | 12.5–14px |

قواعد بدون تغییر از CLAUDE.md:

- هیچ `text-[Npx]` یا `font-[N]` نوشته نشود — CI رد می‌کند.
- ارتفاع کارت KPI: `min-h-*` نه `h-[176px]` ثابت.
- عرض Popover/Overlay: `w-full max-w-[Npx]`.
- ارتفاع پنل نمودار: پلکانی responsive مثل `h-[320px] sm:h-[420px] lg:h-[500px]`.
- جهت‌ها فقط logical: `ms-/me-/ps-/pe-/start-/end-/border-s/e`.
- رنگ فقط از توکن‌های `n-*`؛ رنگ خام نیازمند `dark:` است و در lint رد می‌شود.

نتیجه: §۷ سند اصلی **باطل** و این جدول جایگزین آن است.

## اصلاح ۲ — نگاشت واقعی رجیستری shadcn (جایگزین §۱۵ سند اصلی)

### موجودی فعلی پروژه (نصب نشود — هست)

`chart` ،`table` ،`skeleton` ،`sheet` ،`drawer` ،`command` ،`tabs` ،`card`،
`dropdown-menu` ،`alert` ،`alert-dialog` ،`tooltip` ،`select` ،`popover`،
`calendar` + `jalali-picker` (اختصاصی)، `toggle-group` ،`badge` ،`avatar`.
وابستگی `@tanstack/react-table` نیز از قبل در `package.json` هست.

### کسری‌ها — دستور نصب دقیق

```bash
bunx shadcn@latest add sidebar empty spinner field item kbd
```

| کامپوننت | مصرف در داشبورد |
|---|---|
| `sidebar` | پوسته ناوبری §۳ — حالت `collapsible="icon"` built-in دارد |
| `empty` | تمام Empty stateهای §۸ و §۱۱ — به‌جای ساختن دستی |
| `spinner` | حالت Updating در §۱۱ («پیشرفت ظریف بدون حذف محتوا») |
| `field` | فیلترهای هدر و فرم‌های Sheet موبایل |
| `item` | ردیف‌های «موارد نیازمند اقدام» — ساختار icon/title/meta/action آماده |
| `kbd` | نمایش میانبرها در Command bar و Tooltipها |

### بلوک‌های مرجع (کد نمونه رسمی — کپی و بومی‌سازی)

| نیاز سند اصلی | بلوک رجیستری | نکته پیاده‌سازی |
|---|---|---|
| ترکیب کلی داشبورد | `@shadcn/dashboard-01` | sidebar + chart + data table در یک بلوک |
| سایدبار آیکون‌شونده §۳ | `@shadcn/sidebar-07` | دقیقاً «collapses to icons»؛ فقط RTL و گروه‌بندی فارسی |
| گروه‌های جمع‌شونده ناوبری | `@shadcn/sidebar-02` | الگوی Collapsible sections |
| نمودار اصلی تب‌دار §C | `@shadcn/chart-area-interactive` | تب/سوییچ سری + بازه زمانی built-in |
| نمودار ستونی تعاملی | `@shadcn/chart-bar-interactive` | برای تب «انتشارها» |
| Tooltip فارسی نمودار | `@shadcn/chart-tooltip-formatter` | الگوی formatter — اعداد فارسی و تاریخ جلالی این‌جا تزریق شود |
| جدول انتشار §D | `@shadcn/data-table-demo` | TanStack: sort/filter/visibility/pagination/selection — تمام معیارهای پذیرش §D را پوشش می‌دهد |
| تقویم فارسی | `@shadcn/calendar-hijri` | نمونه رسمی Persian calendar — با `jalali-picker.tsx` موجود مقایسه و بهترین ادغام شود |

بلوک‌های داخلی `@nashrino/*` سند اصلی معتبرند؛ هر کدام باید روی بلوک مرجع بالا
ساخته شوند، نه از صفر.

## اصلاح ۳ — تحلیل شکاف لایه داده (فاز ۰٫۵ — جدید و مسدودکننده)

سند اصلی UI را کامل مشخص کرده اما هرگز نپرسیده بک‌اند چه می‌دهد. قبل از شروع فاز ۱،
یک ممیزی یک‌روزه API با این خروجی الزامی است:

| نیاز UI | منبع داده لازم | وضعیت (پر شود) |
|---|---|---|
| ۴ KPI × فیلتر workspace/بازه/شبکه | endpoint تجمیعی با سه پارامتر فیلتر | ☐ موجود / ☐ نیازمند ساخت |
| مقایسه با دوره قبل | همان endpoint با بازه قبلی یا فیلد delta | ☐ |
| سری زمانی نمودار به تفکیک شبکه | endpoint سری زمانی group-by platform | ☐ |
| موارد نیازمند اقدام (۵ گروه) | union از publish failures / pending reviews / connection errors / unanswered messages / campaign delays | ☐ |
| جدول انتشار (۸ ستون + live) | list endpoint با sort/filter/pagination + کانال realtime موجود | ☐ |
| «آخرین انتشار موفق» هر حساب | فیلد یا query روی جدول publish log | ☐ |
| پیشرفت و مهلت کمپین | مدل Campaign با target/progress/deadline | ☐ |
| نقش کاربر برای اولویت‌بندی | session role — سرور منبع حقیقت | ☐ |

**قاعده تصمیم:** هر ردیفی که «نیازمند ساخت» شد، قبل از فاز UI مربوطه باید issue
بک‌اند باز و تخمین بخورد. اگر جمع تخمین بک‌اند از تخمین UI بیشتر شد، ترتیب فازها
بر اساس آمادگی داده بازچینی می‌شود — نه ترتیب سند.

## اصلاح ۴ — تفکیک MVP از Deferred (بازنویسی دامنه §۱۶)

سند اصلی همه ۸ فاز را هم‌وزن ارائه کرده بود؛ در عمل یعنی هیچ‌کدام تمام نمی‌شود.
دامنه‌بندی نهایی:

### MVP — بدون این‌ها انتشار نداریم

- فاز ۰ (baseline + flag) و فاز ۰٫۵ (ممیزی داده — اصلاح ۳)
- فاز ۱: هدر + فیلترها در URL
- فاز ۲: چهار KPI + Status strip
- فاز ۳: نمودار تب‌دار + پنل اقدام
- فاز ۴: جدول انتشار

### نسخه دوم — بعد از تثبیت MVP

- فاز ۵: پنل کمپین و اتصال حساب‌ها (تا آن زمان کارت‌های فعلی می‌مانند)
- فاز ۶: سایدبار جدید + ساده‌سازی Command bar

### Deferred — فقط با سیگنال واقعی کاربر

- فاز ۷ (رفتار نقش‌محور §۹): تا وقتی چند نقش فعالِ واقعی در production نداریم، فقط
  حالت Read-only برای Viewer پیاده می‌شود؛ اولویت‌بندی ویجت per-role حذف.
- Offline-with-cache در §۱۱: به «پیام آفلاین + حفظ آخرین داده در حافظه» ساده می‌شود.
- Windows High Contrast و Zoom 200% از ماتریس QA به چک‌لیست پیش از v2 منتقل می‌شوند.
- شخصی‌سازی ترتیب ویجت‌ها: حذف از دامنه فعلی (خود سند هم اذعان داشت).

## اصلاح ۵ — سنجش با خط مبنا (جایگزین اهداف §۱۸)

اهداف درصدی سند اصلی («کاهش ۴۰٪...») بدون اندازه‌گیری مبنا ادعاپذیر نیستند. روال:

1. در فاز ۰، ده Event پیشنهادی §۱۸ روی **داشبورد فعلی** فعال می‌شود.
2. دو هفته داده مبنا جمع می‌شود (زمان یافتن انتشار ناموفق، کلیک تا Retry، نرخ ورود
   به صف انتشار، misclick روی عناصر غیرفعال).
3. اهداف درصدی **بعد از** دیدن مبنا نهایی می‌شوند. تا آن زمان فقط دو هدف مطلق
   معتبر است: «کلیک تا Retry ≤ ۲» و «Misclick روی کنترل غیرفعال = صفر».

## اصلاح ۶ — موضع طراحی (چیزی که سند اصلی نداشت)

سند اصلی یک spec مهندسی بی‌نقص اما بی‌چهره بود. سه تصمیم هویتی که این داشبورد را
«نشرینو» می‌کند، نه یک admin panel عمومی:

1. **اعداد قهرمان صفحه‌اند.** در محصولی که کارش سنجش محتواست، عدد KPI تنها عنصر
   با وزن بصری سنگین است: `text-2xl font-extrabold tabular-nums` با ارقام فارسی،
   و همه‌چیز اطرافش (label ،sparkline ،delta) عمداً کم‌صدا. بنفش برند فقط دو جا:
   اقدام اصلی و وضعیت انتخاب‌شده — همان قاعده §۶، اما به‌عنوان امضا نه محدودیت.
2. **جهت روایت راست‌به‌چپ است، همه‌جا.** نمودارها هم: محور زمان از راست (قدیم) به
   چپ (جدید) — با `reversed` در Recharts. Tooltip و legend فارسی با تاریخ جلالی
   (بلوک `chart-tooltip-formatter` + `jalali-picker` موجود). این تنها داشبورد
   ایرانی این رده خواهد بود که نمودارش هم RTL فکر می‌کند.
3. **سکوت یعنی سلامت.** وقتی «موارد نیازمند اقدام» خالی است، پنل جمع می‌شود به یک
   خط سبز آرام («همه‌چیز در وضعیت عادی است») — نه یک Empty state هم‌قد پنل پر.
   فضای آزادشده را نمودار می‌گیرد. داشبوردِ بی‌مشکل باید *احساس* بی‌مشکلی بدهد.
